# Zustandsmaschine 0, PIO-Programm = blink, Frequenz = 2029,
# set-Mapping mit 1 Pin: GPIO25 => eingebaute LED
sm = StateMachine(0, blink, freq=2029, set_base=Pin(25))

sm.active(1)               # Zustandsmaschine starten

sm.active(0)               # Zustandsmaschine stoppen

Interessant ist der Aufruf von sm.exec(), mit dem sich eine Instruktion direkt an die Zustandsmaschine bertragen lsst. Im vorliegenden Fall schaltet die Instruktion die LED wieder aus, bevor das Hauptprogramm die Zustandsmaschine deaktiviert.

Beispiel Ampelsteuerung

Um das Thema PIO praktisch durchzuspielen, bieten sich vor allem Beispiele an, die weder zu trivial noch zu komplex sind. Eine mgliche Option besteht aus dem Simulieren einer Verkehrsampel, wobei natrlich auch andere Signalanlagen in Frage kmen.

In Deutschland haben Ampelsteuerungen bekanntlich vier verschiedene Phasen:

  1. Rot-Phase: Verkehrsteilnehmer mssen warten.
  2. Rot-Gelb-Phase: Verkehrsteilnehmer mssen bis zur Grn-Phase warten.
  3. Grn-Phase: Verkehrsteilnehmer knnen fahren.
  4. Gelb-Phase: Verkehrsteilnehmer mssen anhalten.
  5. zurck zu 1.

Die Phasen Grn und Rot sind individuell konfigurierbar, etwa abhngig von Verkehrsaufkommen. Die Dauer der Gelbphasen (Gelb und Rot-Gelb) ergibt sich aus der am Aufstellungsort erlaubten Hchstgeschwindigkeit.

Gesucht: ein Pico-Programm, das diese Steuerung implementiert. Die Implementierung erfolgt ber C beziehungsweise C++ und PIO-Assemblerinstruktionen. Mgliche Variante: Statt C beziehungsweise C++ liee sich auch MicroPython verwenden.
Es soll mglich sein, die Rot- beziehungsweise Grnphase von auerhalb des PIO zu konfigurieren.

Die Ampel-Schaltung

Die verschiedenen Leuchten der Ampel ersetzt die sehr einfache Beispielschaltung durch LEDs mit Anschluss an GPIO-Pins des Pico. Diese Schaltung lsst sich spter erweitern, etwa durch einen Schalter, der die Ampel auer Betrieb nimmt oder wieder aktiviert. Auf einem Pico stehen zwei PIO-Blocks mit je 4 Zustandsmaschinen zur Verfgung, sodass ebenfalls eine Schaltung mehrerer Ampeln mglich wre.
Insgesamt ist die Schaltung also sehr einfach. Die LEDs sind an GPIO17 (weisse LED oder andere Empfnger), GPIO18 (rote LED), GPIO19 (gelbe LED), und GPIO20 (grne LED) ber jeweils einen 220 Ohm Widerstand und an GND angeschlossen.

Die eigentliche Steuerungsaufgabe fr die vier Ampelphasen bernimmt ein PIO-Programm, dem das Hauptprogramm zu Beginn die Lnge der Rot- beziehungsweise Grnphase sowie ein Bit-Pattern bergibt. Die zwei Gelbphasen (Gelb, Rot-Gelb) dauern jeweils wenige Sekunden. Da die LEDs in der Reihenfolge Rot, Gelb, Grn angeschlossen sind, ergeben sich folgende Patterns. Das Bit fr die frei verwendbare Ausgabe an GPI017 fehlt in folgenden Bit-Sequenzen:

Phase 1 (Rot) lsst sich durch eine Bitsequenz wie 1-0-0 reprsentieren (rote LED an, gelbe LED aus, grne LED aus).

Phase 2 (Rot-Gelb) entspricht dann der Bitsequenz 1-1-0.

Phase 3 (Grn) lautet 0-0-1.

Phase 4 (Gelb) entspricht 0-1-0.

Die GPIO-Pins fungieren im PIO-Programm demzufolge als Ausgabe-GPIOs. Damit dies funktioniert, bentigen wir fr das entsprechende I/O-Mapping, konkret fr das Output-Mapping, benachbarte GPIOs. In dem Mapping sind GPIO-Pins 17, 18 (rot), 19 (gelb), 20 (grn) im Einsatz, wobei GPIO17 mit jeder Rot-Phase ein HIGH-Signal ausgibt. Das knnte man zur Ansteuerung einer weiteren LED benutzen oder fr andere Zwecke. Die Bit-Sequenzen von jeweils 4 Bits legt die sm (state machine) per out-Kommando an die LEDs. Fr einen vollstndigen Durchlauf aller vier Phasen sind somit 16 Bit notwendig. Da die TX-FIFO mit 32 Bit arbeitet, bergeben wir das um einen kompletten Ampelzyklus verdoppelte Pattern als 32-Bit-Wort an das OSR (Output-Shift-Register). Bis das OSR leer ist, vergehen demzufolge zwei vollstndige Iterationen durch alle vier Ampelphasen. Da der Entwickler das Pattern nach Belieben definieren kann, lsst sich die Signalisierung am GPIO17 beliebig einstellen, etwa HIGH-Signal in jeder Rot-Phase oder High Signal zu Beginn aller Phasen.

Das Pattern erthlt das Programm vom C-Hauptprogramm, das zuvor die Zeit fr die Rot- und Grnphase an die Zustandsmaschine bergibt:

        pull       ; Zeit aus der TX-Queue ins OSR holen 
        mov x,osr  ; und ins Register x
        mov isr,x  ; sowie ins ungenutzte ISR 
                   ; kopieren
        pull       ; Pattern von CPU-Core aus 
                   ; der TX-Queue ins OSR holen
        mov y,osr  ; und zustzlich in y speichern

Weil das Programm nach je zwei Ampelzyklen das OSR (Output-Shift-Register) komplett leer gerumt hat, ist das Pattern auch in y abgelegt, um es erneut ins OSR laden zu knnen:

        jmp !OSRE cont  ; OSR != EMPTY 
                        ; => weiter zu cont
        mov osr,y       ; Sonst: OSR neu laden
     cont:    
        ....

Ebenfalls wird das im Programm unbenutzte ISR (Input-Shift-Register) als Backup fr die Zahl der Zeitschleifen benutzt:

        mov x,isr              ; ISR enthlt Schleifenzahl fr x
        ...
     lgreen:
        nop [31]
        jmp x-- lgreen [31]    ; hier wird x in jeder           
                               ; Schleife dekrementiert

Das PIO-Assembler-Programm schiebt das OSR immer um 4 Bit nach rechts, um mit den Bits die 4 Ausgabe-Pins anzusteuern:

        out pins, 4    ; Schiebe 4 Bits aus dem OSR 
                       ; nach rechts zu den LEDs bzw. GPIOs

So entstehen nacheinander und zyklisch die vier Ampelphasen (rot, rotgelb, grn, gelb).

    mov x,osr     ; und in x speichern

% c-sdk {

// Hilfsfunktion, um die Zustandsmaschine zu konfigurieren 

void trafficlight_program_init(PIO pio, uint sm, uint offset, 
                               uint pin, uint pincount, uint freq) {

   for (uint i = 0; i < pincount; i++) {
        pio_gpio_init(pio, (pin+i) % 32);   // initialisieren aller Pins
   } 

   // pins als Ausgabe-Pins (true) festlegen
   pio_sm_set_consecutive_pindirs(pio, sm, pin, pincount, true);

   // Default Configuration holen
   pio_sm_config c = trafficlight_program_get_default_config(offset);

   // Die vier Pins definieren das out-Mapping:
   sm_config_set_out_pins(&c, pin, pincount);

   // div <= 65535 (= 2^16-1) - wird hier nicht geprft
   float div = (float)clock_get_hz(clk_sys) / freq;

   // Jetzt Clock Divider bergeben
   sm_config_set_clkdiv(&c, div);

   // Wir kombinieren beide FIFOs zu einer TX_FIFO;
   sm_config_set_fifo_join(&c, PIO_FIFO_JOIN_TX);

   // Rechts-Schieber, kein auto-pull, Schwellwert: 32 Bits
   sm_config_set_out_shift(&c, true, false, 32);

   // Zustandsmaschine initialisieren
   pio_sm_init(pio, sm, offset, &c);

   // und starten
   pio_sm_set_enabled(pio, sm, true);
}
%}

Am Ende des Assemblercodes befindet sich die mit % c-sdk { beziehungsweise %} geklammerte C-Funktion void trafficlight_program_init(...), die der PIO-Assembler in die generierte xxxxpio.h-Datei bernimmt.
Diese Hilfsfunktion hat mehrere Aufgaben:
Zunchst teilt sie den einzelnen GPIOs ber Aufrufe von pio_gpio_init()mit, dass sie unter Kontrolle der PIO stehen.

Die Anweisung

pio_sm_set_consecutive_pindirs()

legt fest, dass die hintereinander folgenden GPIO-Pins als Ausgabe-Pins fungieren.
Zur Konfiguration der Zustandsmaschine bedarf es einer entsprechenden Datenstruktur (offset ist die Anfangsadresse des Programmes):

pio_sm_config c = trafficlight_program_get_default_config(offset);

Nun erfolgt die Festlegung des out-Mappings:

sm_config_set_out_pins(&c, pin, pincount);

Die Zeitdauern fr die verschiedenen Ampelphasen hngen von zwei Faktoren ab:
Der fr die Zustandsmaschine gewhlten Frequenz und den in Schleifen hinzugefgten Delays.
Die Frequenz einer Zustandsmaschine ist mittels des Clock Dividers beeinflussbar. Nimmt man mit 65535 das Maximum, liegt die Frequenz der Zustandsmaschine, wie bereits erwhnt, bei grob 2000 Hz (eigentlich 2029 Hz), was einer Zykluszeit von grob 0,5 Millisekunden entspricht. Mittels der Wahl der Delays in den PIO-Instruktionen und der von aussen bergebenen Verzgerungszeit knnen Entwickler die gewnschten Ampelphasen sehr gut annhern.

Um eine gewnschte Frequenz einzustellen, berechnet die Funktion einen Clock-Divider und konfiguriert damit die Zustandsmaschine:

float div = (float)clock_get_hz(clk_sys) / freq;
sm_config_set_clkdiv(&c, div);

Da das Programm die RX-FIFO nicht nutzt, verbinden wir sie mit der TX-Queue zu einer doppelt so groen TX-Queue, was eigentlich im Ampel-Beispiel nicht ntig ist, sondern lediglich zur Illustration dieser Mglichkeit dient:

sm_config_set_fifo_join(&c, PIO_FIFO_JOIN_TX);

Das OSR (Output-Shift-Register) schiebt die Bitmuster fr die Ampelsignale nach rechts (erstes true), macht dies manuell statt automatisch (false) und hat einen Schwellwert von 32: Das heit, nach geschobenen 32 Bits interpretiert die Zustandsmaschine das OSR als leer:

sm_config_set_out_shift(&c, true, false, 32);

Am Ende wird die Zustandsmaschine entsprechend der Konfiguration initialisiert:

pio_sm_init(pio, sm, offset, &c);

und scharf gestellt beziehungsweise gestartet:

pio_sm_set_enabled(pio, sm, true);

#include <stdio.h>
#include "pico/stdlib.h"
#include "hardware/pio.h"
#include "hardware/clocks.h"
#include "pio_trafficlight.pio.h"


const uint freq =     2029;  // Gewnschte Frequenz der Zustandsmaschine
const uint pin      =   17;  // Start-Pin 
const uint pincount =    4;  // Zahl der Ausgabe-Pins
                             //     ==> RIGHT SHIFT ==>
                             // 2 x Ausgabe-Pattern fr GPIOs          
const uint32_t pattern =  0x48634863; // = 0100 1000 0110 0011 ... 

const uint32_t delay = 200;  // initial Zeitdauer fr Gelb-Phasen

int main() {
    setup_default_uart();

    // Wir verwenden pio0
    PIO pio = pio0;
    // Programm zum Programmspeicher hinzufgen => offset
    uint offset = pio_add_program(pio, &trafficlight_program);
    printf("Programm geladen an Adresse %d\n", offset);
    // Frei verfgbare Zustandsmaschine akquirieren
    int sm = pio_claim_unused_sm(pio, true);
    // Initialisieren und Konfigurieren der Zustandsmaschine
    trafficlight_program_init(pio, sm, offset, pin, pincount, freq);
 
    // Delay an TX-FIFO der Zustandsmaschine bergeben
    pio_sm_put(pio, sm, delay);
    // Bisschen warten
    sleep_ms(500); 
    // Bitmuster zum LED-Schalten an Zustandsmaschine bergeben
    pio_sm_put(pio, sm, pattern);
    sleep_ms(1000000); // Lange laufen lassen
    pio_sm_set_pins(pio, sm, 0); // Alle Pins auf Low
    pio_sm_set_enabled(pio, sm, false); // Zustandsmaschine stoppen
}

Das Programm benutzt den Block pio0, fgt das PIO-Programm zu dessen Programmspeicher hinzu, um am Schluss eine freie Zustandsmaschine anzufordern:

PIO pio = pio0;
uint offset = pio_add_program(pio, &trafficlight_program);
int sm = pio_claim_unused_sm(pio, true);

Danach folgt der Aufruf der eben erwhnten Hilfsfunktion:

trafficlight_program_init(pio, sm, offset, pin, pincount, freq);

Erst sendet das Programm die gewnschte Zahl der Warteschleifen zur TX-Queue der Zustandsmaschine:

pio_sm_put(pio, sm, delay);

und danach das Bitmuster fr die Ansteuerung der LEDs:

pio_sm_put(pio, sm, pattern);

Wichtig: Das Bit-Muster ist wegen der Right-Shifts zu den 4 GPIOs in 4er-Gruppen von rechts nach links zu lesen. Die 4er-Gruppen steuern jeweils mit dem LSBit GPIO17 (Kontrollausgabe zur beliebigen Verwendung), GPIO18 (Rot), GPIO19 (gelb) und mit dem MSBit GPIO20 (Grn). Das Muster 0x4863.... (= binr 0100 1000 0110 0011) bedeutet also die Abfolge:

  1. 0011 => GPIO17 = HIGH, RED = HIGH, YELLOW = LOW, GREEN = LOW
  2. 0110 => GPIO17 = LOW, RED = HIGH, YELLOW = HIGH, GREEN = LOW
  3. 0001 => GPIO17 = LOW, RED = LOW, YELLOW = LOW, GREEN = HIGH
  4. 0100 => GPIO17 = LOW, RED = LOW, YELLOW = HIGH, GREEN = LOW
  5. ....

Das Programm wartet nach den put()-Operationen fr eine festgelegte Zeit von einer Million Millisekunden (etwa 16 Minuten), whrend die PIO die Ampel steuert: 

sleep(1000000);

Nach dieser Wartezeit setzt es alle Ausgabe-Pins auf Low und deaktiviert die Zustandsmaschine wieder:

pio_sm_set_pins(pio, sm, 0); 
pio_sm_set_enabled(pio, sm, false);
PIO-Programmoptimierung

Mit Instruktionen sollten Entwickler gut haushalten, weil fr alle vier Zustandsmaschinen eines PIO-Blocks nur insgesamt 32 Slots im gemeinsamen Programmspeicher verfgbar sind.

Wie also liee sich das Programm optimieren? Im PIO-Teil der Ampelsteuerung durchluft das Assemblerprogramm die vier Phasen einer Ampel. Eigentlich lsst sich das alles als zweimal zwei Phasen betrachten: je eine lange Phase (rot oder grn), gefolgt von einer kurzen Phase (gelb oder rot & gelb). Somit ist folgende gekrzte Version mglich: 

.program trafficlight
    pull             ; Zeit von CPU-Core holen
    mov x,osr        ; und in x speichern
    mov isr,x        ; sowie im ungenutzten ISR 
    pull             ; Pattern von CPU-Core holen
    mov y,osr        ; und in y speichern
.wrap_target
    jmp !OSRE cont   ; OSR != EMPTY => weiter bei cont
    mov osr,y        ; Sonst: OSR leer => neu aufladen
cont:
;                    ROT oder GRN
    mov x,isr        ; ISR enthlt Schleifenzahl fr x
    out pins,4       ; 4 Bits aus OSR zur Ausgabe
lredgreen:
    nop [31]
    jmp x-- lredgreen [31]
;                    ROT-GELB oder Gelb
    mov x,isr        ; ISR enthlt Schleifenzahl fr x
    out pins,4       ; 4 Bits aus OSR zur Ausgabe
lyellowred: 
    nop [10]
    jmp x-- lyellowred [10] 
.wrap

Nachteil:

Jetzt dauern Rot- und Grnphase sowie Rotgelb- und Gelbphase jeweils gleich lange, aber das ist leicht zu verschmerzen.
Fazit

Die Programmierbare Ein-/Ausgabe des Raspberry Pi Pico, kurz PIO, bietet einen echten Mehrwert fr Entwickler. Natrlich lieen sich fr den gleichen Zweck FPGAs nutzen, aber die sind teuer und bentigen wesentlich mehr Erfahrung in digitaler Schaltungstechnik. Gegenber dem sonst blichen Bit-Banging hat PIO den Vorteil, dass Zustandsmaschinen zeitlich genau arbeiten und nicht die beiden CPU-Kerne belasten. Sie offerieren einen kleinen, aber leistungsfhigen Satz von neun Instruktionen, erlauben dabei die Umsetzung auch komplexerer I/O-Schnittstellen, wofr sich im Internet bereits zahlreiche Beispiele finden. Das vorgestellte Beispiel ist bewusst einfach gehalten, sollte aber einen Einblick in die Mglichkeiten der PIO zeigen. Wie heit es in der Werbung doch so schn: Entdecke die Mglichkeiten!
